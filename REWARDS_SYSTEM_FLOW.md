-- Total rewards distributed today
SELECT
  COUNT(*) as orders,
  SUM(total_points) as total_points,
  SUM(itc_bonus) as total_itc
FROM order_rewards
WHERE DATE(awarded_at) = CURRENT_DATE
AND status = 'awarded';

-- Referral program stats
SELECT
  COUNT(DISTINCT referrer_id) as active_referrers,
  COUNT(*) as total_referrals,
  SUM(referrer_reward_points) as points_distributed,
  SUM(referrer_reward_itc) as itc_distributed
FROM referral_transactions
WHERE status = 'completed'
AND DATE(created_at) >= CURRENT_DATE - INTERVAL '30 days';

-- Tier distribution
SELECT
  user_tier,
  COUNT(*) as users,
  AVG(total_points) as avg_points_per_order
FROM order_rewards
WHERE status = 'awarded'
GROUP BY user_tier
ORDER BY user_tier;

-- Failed rewards needing attention
SELECT
  order_id,
  user_id,
  order_total,
  metadata->>'error' as error_reason,
  created_at
FROM order_rewards
WHERE status = 'failed'
ORDER BY created_at DESC;

## Key Concepts

### Atomicity
All reward operations are atomic - either all succeed or all fail. No partial rewards.

### Idempotency
The system prevents duplicate rewards through unique constraints. Safe to retry.

### Auditability
Every reward operation is logged in audit_logs table with full context.

### Scalability
- Indexed queries for fast lookups
- Batch processing support
- Asynchronous reward processing capability

### Security
- RLS at database level
- JWT authentication at API level
- Role-based access for admin operations
- Input validation throughout
